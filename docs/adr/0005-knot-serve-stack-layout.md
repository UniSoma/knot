# knot serve adopts a single-column stack layout

The browser-side `knot serve` panel renders tickets as a single column of three collapsible groups (In progress / Ready / Blocked) with click-row-to-expand detail — the same shape `knot ready` produces in the terminal and `knot.el` produces in Emacs. We picked this after prototyping three competing variants (Stack / Workbench / Dashboard) in `prototype/serve/` and finding that (1) the Stack carries zero relearning cost for users coming from the CLI or Emacs surface, and (2) only the Stack survives a ~380px VSCode Webview without collapsing — the Workbench at narrow widths *becomes* a worse Stack (its right metadata rail folds out at ≤1100px and the three panes stack at ≤720px), and the Dashboard loses its at-a-glance rhythm below ~900px.

The cost is that wide-screen users don't get a dedicated metadata rail or a flagship "next ticket" hero out of the box — both shapes the Workbench and Dashboard variants made compelling. We're accepting that cost because both can be added back as progressive enhancements on top of the Stack base (a `?id=` deep-link with an expanded detail pane is a sidecar, not a rewrite), whereas the inverse path — start with the Workbench or Dashboard shape and retrofit narrow-viewport — is what the prototype proved doesn't work.
