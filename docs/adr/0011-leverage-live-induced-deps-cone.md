# Leverage scores a ticket by its live-induced forward deps cone

The three *live* listing commands (`list`, `ready`, `blocked`) gain a **`LEV`** column and a `leverage` integer in `--json`. Leverage is the count of *live* tickets that transitively depend on a row through `:deps` — the size of its directed forward unblocking cone, the row itself excluded — computed over the **live-induced** deps subgraph. It is always computed and always shown; in v1 it never sorts, never gates, and never appears on `closed`. The number answers one question: *if I close this ticket, how much downstream work stops waiting?* High leverage marks the keystones the open backlog is stacked on, so reading the column top-down is how a human (or agent) picks what to dissolve first — and dissolving keystones is what de-entangles the deps graph.

Three properties define the metric, each a deliberate cut against a plausible alternative. **Directed and `:deps`-only**, not undirected centrality: only `:deps` encodes "waiting" (per `CONTEXT.md`, `:parent` is composition and `:links` is see-also, and neither gates readiness), so closing a node only frees work that was sequenced behind it. An undirected degree would rank a sprawling umbrella or a dense link-cluster as "high-leverage" when closing it frees nobody. **Total transitive cone**, not marginal/immediate unblock: the question is structural keystone-ness ("what ultimately waits behind this"), not scheduler throughput ("what becomes `ready` the instant I close this"). The marginal count is a different, legitimate scalar — it just answers a question nobody asked here. **A plain count**, not a weighted blend: priority and value stay separate axes; folding them in would produce an opaque, untunable score whose ranking nobody could trust or explain.

The crux is the **live-induced traversal**. Closed tickets are not merely excluded from the tally — they are *non-conductive*. Given `A → B → C` (A deps B, B deps C) with B closed, A no longer waits on C in any real sense: B is terminal, so it severs the chain, and A is itself `ready`. A status-blind walk would still credit C with A — a phantom dependent. Building the cone over the subgraph *induced on live tickets* keeps leverage exactly equal to the live "waiting" structure that `ready`/`blocked` already act on.

This is the precise *opposite* of ADR 0010's corpus choice, and deliberately so. `--closure` is computed over the **full corpus** because closure answers a *membership* question — "what is related to this seed?" — which is status-agnostic and is only graph-faithful if archived nodes still conduct (a closed child must not block the walk to its open grandchildren). Leverage answers a *live-work* question — "what live work is gated by this?" — which is only faithful if closed nodes *stop* conducting. Same graph, opposite traversal scope, because the questions are opposite. The corpus scope follows the question, not a convention.

The surface is intentionally minimal: no flag, no sort, an always-on column, and an always-on `--json` field. Always-on JSON does **not** reverse ADR 0010's rejection of `--json` pollution — that rejection targeted per-*query metadata* (`closure_via`, `closure_distance`: annotations explaining *why a row matched this filter*). Leverage is the opposite kind of thing: an **intrinsic property of a live ticket**, as stable and query-independent as `priority` or `status`. Adding an intrinsic field is additive and safe (consumers ignore unknown keys), and it is what keeps the deferred downstream surfaces — knot.el heat-coloring, a `knot serve` node-colored graph — reachable without later re-introducing a flag.

## Considered options

- **Undirected centrality / degree over `:parent` + `:deps` + `:links`** — rejected. Mirrors `closure-set`'s axes, but counts org hubs (a big umbrella, a dense link cluster) that closing frees nobody. Conflates composition and see-also with sequencing; only `:deps` carries "waiting."
- **Marginal / immediate unblock** (count that becomes `ready` the instant the row closes) — rejected as the v1 metric. A scheduler's throughput signal, not a structural keystone signal. Legitimate, but a different question; can land later as a sibling scalar.
- **Priority / value-weighted composite** (`w₁·leverage + w₂·priority + …`) — rejected. Opaque and untunable; nobody can explain a blended rank. Priority stays an orthogonal axis.
- **Status-blind ("traverse-through-closed") walk** — rejected. Produces phantom overcount: credits a keystone for dependents that a since-closed intermediary already released. Breaks the "what live work is gated" semantics.
- **Full-corpus computation (mirroring `closure-set`)** — rejected. Would count archived/closed dependents that are already done, inflating the number with work that no longer waits. Faithfulness for a live-work question requires the live-induced subgraph; see the corpus-scope contrast above.
- **Standalone `knot leverage` command** — rejected, per ADR 0010's precedent. A column on the existing commands composes for free with `--type` / `--tag` / `--closure` / `--limit` (rank the `ready` *bugs*; read leverage across a `--closure` cluster). A separate command would re-implement that composition.
- **`--leverage` opt-in flag gating the column and JSON field** — rejected in favor of always-on. The number is intrinsic and cheap (the corpus is already in memory for every listing), and gating the JSON field would starve the downstream el/serve coloring surfaces of their only data source.
- **Always-on `LEV` column vs. the lean-output value** — the closure feature deliberately added *no* column (ADR 0010 rejected `VIA`/`DIST`). Leverage is admitted anyway because it is an unambiguous per-row scalar — none of the multi-path / multi-seed ambiguity that sank a closure column — and a 3-char integer is cheap on the narrowest terminal.
- **Leverage on `closed`** — rejected. A closed ticket is not live; its cone is ~0 by construction. An all-zero column is noise.
- **`--sort leverage` / a general `--sort <key>` framework** — deferred to v2. Purely additive; inventing a sort grammar now is speculative scope. v1 presents the number and lets the reader rank by eye.
- **Depth- or edge-weighting the cone** — deferred. Count first; weight only if the plain count is shown to mislead.

## Consequences

- A new primitive lands in `query.clj` — the transitive, *live-only* sibling of `blocking` (which returns *direct* dependents). It builds the reverse-`:deps` adjacency over live tickets, BFS-walks from each row with a visited-set cycle guard, and drops broken refs — the same conventions already used by `closure-set` / `dep-tree` / `project-cycles`.
- `list`, `ready`, and `blocked` render a `LEV` column and add a `leverage` integer to `--json`. `closed`, `show`, `dep tree`, `delete`, `check` are untouched. Default human output gains one narrow column; default JSON gains one additive, intrinsic field.
- `CONTEXT.md` gains a **Leverage** glossary entry under "Ticket relationships," with `_Avoid_` steering away from "centrality" / "degree," "priority," "marginal unblock," and "heatmap."
- v2 surfaces are additive and need not revisit this ADR: `--sort leverage` (ranking), knot.el row heat-coloring, and a `knot serve` node-colored dependency-graph view — all consume the `leverage` JSON field this ADR establishes.
- Per the AGENTS.md skill-sync rule, `.claude/skills/knot/SKILL.md` must document the `LEV` column and the `leverage` JSON field in the same commit that ships them.
