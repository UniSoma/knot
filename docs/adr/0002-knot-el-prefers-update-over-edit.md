# knot.el prefers `knot update` + capture buffers over `knot edit` + emacsclient

knot deliberately splits its mutation surface: `knot edit` is the interactive editor-spawning path for humans; `knot update` is the non-interactive flag-driven path for agents. A future reader of knot.el — given that the user *is* a human — would reasonably expect the package to wire `EDITOR=emacsclient` and shell out to `knot edit` for every mutation.

We don't. knot.el routes almost every mutation through `knot update` with explicit flags: status, priority, mode, type, tags, assignee, parent, individual acceptance criteria, deps, and links each map to a single `--flag value` call from a transient. Long-form fields (description, design, body, notes) pop a markdown-mode capture buffer that pipes its content via `knot update --description ...` (or stdin for `add-note`) on `C-c C-c`. `knot edit` stays available as a capital-`E` escape hatch in the show buffer for arbitrary structural edits.

The reasoning is that `knot edit` is whole-file editing — appropriate when a human at a terminal wants to restructure a ticket, wrong when an Emacs UI wants to bump priority without popping a buffer at all. knot.el's editing UX is granular by design: each transient action is a single CLI call with one flag, refresh, done. Granular `knot update` calls also keep knot.el's mutations close to what the agent-facing CLI surface already does, so changes to the contract show up in one place. The trade-off is more glue code in knot.el than a single `knot edit` call would need — paid because the alternative (a buffer pop for every priority bump) would defeat the magit-style fluidity we're going for.
