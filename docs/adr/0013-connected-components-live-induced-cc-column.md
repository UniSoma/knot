# Connected components label a ticket by its live-induced cluster

The three *live* listing commands (`list`, `ready`, `blocked`) gain a leading **`CC`** column and a `cc` field in `--json`. CC marks which **connected component** of the **live-induced** graph each ticket belongs to, over all three relationship axes (`:parent` ∪ `:deps` ∪ `:links`, undirected). It completes the derived-column family beside leverage (`LEV`, ADR 0011) and coupling (`CPL`, ADR 0012): where coupling scores a ticket's *local* 1-hop entanglement, CC names the *whole island* it sits on. The number answers a different question — *which tickets form one body of work?* — so two rows sharing an ordinal are reachable from each other through live relationships, and the eye groups them on sight even when filtering or sorting has scattered them down the list. It is the at-a-glance "these belong together" signal that neither leverage nor coupling gives: those are per-row scalars; CC is a partition.

Four properties define the label, each a deliberate cut against a plausible alternative. **Live-induced scope, like LEV/CPL and against `--closure`.** Membership flows only through live tickets; a closed ticket is non-conductive — not a node, and edges incident to it are dropped — so a cluster joined only through a closed bridge *splits*. This is the family's shared stance: the column reflects the *current* backlog's shape, the only structure you can act on. It deliberately diverges from `--closure`, which is corpus-wide single-seed membership (a closed ticket still conducts there) — the asymmetry is intended, not an oversight: `--closure` answers "everything transitively related to *this seed*, ever"; CC answers "the live islands of the *whole backlog*, now." **All three axes, undirected** — the same axis set as `--closure`, because component identity is a pure reachability question and every relationship is a connection regardless of direction or kind. (Contrast coupling, which prunes `:parent` and direction because its question is cognitive load, not reachability.) This accepts epic-driven mega-blobs: an umbrella with many children forms one large component, which is faithful — they *are* one body of work. **Filter-independent membership and size.** The partition is computed once over all live tickets; `--tag`/`--type`/`--limit` never change a row's component or its ordinal. A filtered view is a *window* onto a fixed global partition, not a re-partition of the visible rows — so the same ticket reads `1` whether you ran `list` or `list --tag p0`. **A throwaway global ordinal, not a stable id.** Only components with **≥2 live members** get a number; **singletons render `-`**. Numbers are assigned **size-descending** (largest live component = `1`), ties broken by **min member id**, numbered **globally** over all live multi-member components. The ordinal is a within-snapshot reading aid, not an identifier: it is recomputed every invocation and a filtered view may show non-contiguous numbers (`1, 3, 4`) because numbering is never per-view. Closing a ticket or adding an edge can renumber everything — and that is fine, because the number's only job is to group the rows *in front of you right now*.

Two surface choices distinguish CC from its LEV/CPL siblings. **It is the leading column, before `ID`** — `CC ID STATUS PRI MODE TYPE ASSIGNEE AGE [AC] [CHLD] [LEV] [CPL] TITLE` — because a cluster label is a *row-grouping* key, and grouping keys read best at the left margin where the eye scans for sameness; LEV/CPL are per-row scores that belong in the metric cluster before `TITLE`. **Presence-gating is stricter than LEV/CPL's.** LEV/CPL gate on key-presence (always present on the three listings). CC is present **iff at least one visible row carries a real ordinal** — if every visible row is a singleton, the column vanishes. This is deliberate: because the `cc` JSON field is emitted on *every* row (often `null`), a key-presence gate would surface an all-`-` column on any singleton-only view, which is pure noise. The text column earns its space only when it can actually group something.

The JSON contract mirrors the family's always-on shape with one twist: `cc` is emitted on **every** list/ready/blocked row — an integer ordinal for multi-member rows, **`null` for singletons** — so consumers read a uniform shape and never branch on key-presence (matching how `leverage`/`coupling` always appear). As with LEV/CPL, this does not reopen ADR 0010's rejection of `--json` pollution: that targeted per-*query* metadata explaining why a row matched *this* filter. A ticket's component is an **intrinsic property of the live corpus**, as query-independent as `leverage` — the same whether you ran `list`, `list --tag x`, or `ready`. The field is additive and safe (consumers ignore unknown keys), and it is what keeps the deferred downstream surfaces — knot.el component-coloring, a future `--component N` filter — reachable without a flag.

## Considered options

- **Corpus-wide components (mirroring `--closure`'s scope)** — rejected. Closed tickets would conduct, fusing live islands through long-dead bridges and reporting structure you cannot act on. The live-induced scope keeps CC equal to the *current* backlog's shape, consistent with LEV/CPL. The asymmetry with `--closure` is documented rather than removed: the two answer different questions (whole-backlog islands now vs. one-seed transitive membership ever).
- **Coupling's axis set (`:deps`+`:links`, `:parent` excluded)** — rejected. Component identity is a reachability question; an undirected `:parent` edge genuinely connects an umbrella to its children, and excluding it would shatter epics into spurious fragments. The mega-blob is faithful, not a defect.
- **A stable per-component identifier (e.g. min member id, or a persisted label)** — rejected for v1. A stable id is a heavier contract (persistence, renumbering policy on merge/split) for a column whose job is at-a-glance grouping within one snapshot. The throwaway ordinal is the minimum that groups the visible rows; a stable handle is deferred until a use (cross-invocation references, `--component` filter) demands it.
- **Number singletons too (e.g. each its own ordinal)** — rejected. A label every row carries uniquely conveys no grouping; `-` for singletons reserves the number for its only purpose — marking *shared* membership.
- **Per-view renumbering (contiguous `1..k` over visible components)** — rejected. It would make a ticket's CC depend on the filter, breaking the "intrinsic property" contract CC shares with LEV/CPL and confusing anyone comparing two views. Global numbering with possible gaps is the honest rendering of a fixed partition seen through a window.
- **Ascending size or creation-order ordinals** — rejected. Size-descending puts the largest, most-attention-worthy cluster at `1`, matching how the column is read (find the big island first). Min-id tie-breaking makes the assignment deterministic and reproducible across invocations on unchanged input.
- **Key-presence JSON gate / key-presence column gate (LEV/CPL parity)** — split. JSON keeps the always-emit shape (uniform, `null` for singletons) for consumer simplicity. The *text column* takes the stricter ordinal-present gate, because an always-on column would render all-`-` on singleton-only views — noise the JSON consumer can filter but the human reader cannot.
- **Placing `CC` in the pre-`TITLE` metric cluster beside LEV/CPL** — rejected. CC is a grouping key, not a per-row score; grouping keys read best at the left margin. LEV/CPL stay in the metric cluster.
- **Color-coding components in v1** — deferred. A plain digit ships first, mirroring LEV/CPL; knot.el component-coloring consumes the `cc` JSON field this ADR establishes.
- **CC on `closed` / `show`** — rejected. `show` is a single ticket (no partition to read) and `closed` is the archive (every component is historical); both stay byte-unchanged, as do all touched-mutator outputs.

## Consequences

- A new primitive lands in `query.clj` (`connected-components`): it builds the undirected `:parent` ∪ `:deps` ∪ `:links` adjacency *restricted to live tickets*, floods each unvisited node to collect its component, then assigns size-descending global ordinals (ties by min member id) to the ≥2-member components, returning an `id → ordinal` map (singletons/closed absent → `nil` lookup). Broken refs are dropped and the flood is visited-guarded. `closure-set` is untouched (it stays corpus-wide single-seed for the `--closure` filter).
- `cli.clj` gains `annotate-cc`, which computes the partition once over the full live corpus and attaches `:cc <ordinal-or-nil>` to each row on `list`/`ready`/`blocked`.
- `output.clj` prepends the `CC` column (left-aligned, gated on a visible non-`nil` ordinal), renders `nil`/singleton as `-`, and emits `cc` (integer or `null`) on every list/ready/blocked JSON row. `closed`, `show`, `dep tree`, `delete`, `check` are untouched.
- `CONTEXT.md` gains a **Connected components** glossary entry under "Ticket relationships," with `_Avoid_` steering away from "closure," "coupling," and "stable id."
- Per the AGENTS.md skill-sync rule, `.claude/skills/knot/SKILL.md` documents the `CC` column and the `cc` JSON field in the same commit that ships them.
