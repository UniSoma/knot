# Component filter isolates one live-induced cluster

The three *live* listing commands (`list`, `ready`, `blocked`) gain `--component <id>` — a per-row membership predicate that admits exactly the tickets in the same **connected component** of the **live-induced** graph as the seed `<id>`, over all three axes (`:parent` ∪ `:deps` ∪ `:links`, undirected). It is the action-companion to the `CC` column (ADR 0013): the column *reveals* the live islands at a glance; `--component` *isolates* one island to work on it. Where `--closure` (ADR 0010) is the corpus-wide exploration tool ("everything related to this seed, ever"), `--component` is the live-backlog work tool ("the island this ticket sits on, now") — and the difference is not cosmetic.

`--component` is a distinct flag, not a live-restricted mode of `--closure`, because the two compute over **different graphs**. `closure-set` walks the full corpus, so a *closed* ticket conducts: for live `A` — closed `C` — live `B`, the closure of `A` is `{A, C, B}`, and filtering that result to live rows yields `{A, B}`. The live-induced component of `A`, by contrast, treats `C` as **non-conductive** — not a node, incident edges dropped — so `A` and `B` fall in *different* components and `--component A` returns `{A}`. The two disagree precisely on clusters bridged only through closed tickets. A `--live` modifier bolted onto `--closure` that merely filtered the *result* would therefore report a set that does **not** match the `CC` column it is meant to companion — a lie relative to the column. `--component` must restrict the **traversal** (closed = non-conductive), which is exactly what `connected-components` does and `closure-set` deliberately does not. This is the ADR 0013 asymmetry, now load-bearing for a filter rather than a column.

That different-graph identity is what fixes `--component`'s shape, and the fixity is the feature. **No `--via`.** ADR 0013 fixed component identity at all-axes undirected because reachability is the whole question; a narrowed component would match no `CC` column the user can see, so axis-tuning is meaningless here and is omitted. **Single seed, id-only.** The intent is to isolate *one* island, and any member names the whole island (the column hands you a member id to feed back in), so multi-seed union buys nothing when seeds are co-clustered and blurs the "one island" intent when they are not. The seed is resolved by partial id like `show`/`--parent`/`--closure`, and **never** by ordinal: ADR 0013 makes ordinals throwaway and invocation-unstable, and a bare integer is irreducibly ambiguous with a partial id. **Seed must be live.** A closed ticket is not a node in the live-induced graph, so `--component <closed-id>` is a fail-fast error, not a silent empty: the id resolves to a real ticket but the *component* is undefined, and silently returning nothing would leave the user's "show me this cluster" model broken with no explanation. Falling back to corpus closure is worse still — it is the live-vs-corpus lie this ADR exists to reject.

Composition mirrors `--closure` exactly: membership is computed once over the **full live corpus** (not the display-filtered base), intersected into the existing pipeline *before* `filter-tickets`, with `--limit` applied last. So `list --component X --tag p0` is `(X's live component) ∩ (p0-tagged)` — the component set is filter-independent, the same partition the `CC` column promises. `ready --component X` is the plain intersection "what in this cluster can I start now"; it does not confuse the isolation intent because `ready` is *never* a complete view of anything — `--component` just scopes an already-partial readiness view. The `CC` column is left untouched under `--component`: a multi-member view renders the cluster's (non-contiguous, global) ordinal as a constant column, and a singleton seed shows no column via the existing presence-gate. Suppressing the now-constant column only when `--component` is active would re-couple the column to a specific filter — the exact per-view coupling ADR 0013 rejected — so the constant column stays, earning its space as a cross-reference handle back to the unfiltered list and staying meaningful when an emacs filter sticks across a list→ready switch.

`--component` and `--closure` are **mutually exclusive**: passing both is a fail-fast error. Their intersection would fall out of the pipeline for free, but it answers nothing coherent — "in `X`'s live component *and* in `Y`'s closed-conducting corpus closure" is exotic enough to be, in practice, a user mistake. Erroring is cheap insurance, documents the corpus-tool-vs-live-tool distinction at the point of confusion, and relaxes backward-compatibly if a real intersection use ever appears. The tidy scope rule that results: `--component` rides exactly the commands with a `CC` column (`list`/`ready`/`blocked`), while `--closure` rides all four listing commands including `closed`. `closed --component` is excluded by construction — the archive is all-terminal, so it holds no live components.

## Considered options

- **A `--live` modifier on `--closure` instead of a new flag** — rejected. A modifier that filters `--closure`'s *result* to live rows keeps closed bridges in the membership set (live `A` — closed `C` — live `B` → `{A, B}`), so it reports a set that disagrees with the `CC` column on closed-bridge clusters. Matching the column requires a live-restricted *traversal*, which is a different graph, not a post-filter — so it earns a distinct flag.
- **Accepting an ordinal as the seed (`--component 1`)** — rejected. Ordinals are throwaway and invocation-unstable (ADR 0013); `--component 1` would name a different cluster after any close or edge-add. A bare integer is also ambiguous with a partial id, and there is no way to disambiguate without a second flag for an unstable handle. Id-only loses no expressiveness — any cluster member names the whole cluster.
- **Silent empty result for a closed seed** — rejected. The seed resolves to a real ticket, so this is not a resolution failure; but the component is undefined, and an unexplained empty list leaves the user guessing. A specific error is the honest, actionable failure.
- **Falling back to corpus closure for a closed seed** — rejected. Silently swapping in different graph semantics (corpus, closed-conducting) is the live-vs-corpus lie this whole ADR rejects.
- **`--via` axis tuning on `--component`** — rejected. Component identity is fixed at all-axes (ADR 0013); a narrowed component would match no visible `CC` column. Axis tuning stays a `--closure` affordance.
- **Multi-seed union (comma-split / repeatable, like `--closure`)** — deferred, not killed. The isolation intent and the emacs single-cursor workflow are single-seed; union blurs the "one island" view and breaks the constant-`CC`-column property. Relaxing to union later is additive if an actual ask appears.
- **Suppressing the `CC` column when `--component` is active** — rejected. The column is tautologically constant under `--component`, but suppressing it only there re-couples column presence to a specific filter — the per-view coupling ADR 0013 rejected. The constant column is mildly redundant but honest and useful as a cross-reference.
- **Composing `--component` with `--closure` (intersection)** — rejected for v1. The intersection of a live-island and a corpus closure answers no coherent question and is almost certainly a user mistake; fail-fast is cheaper than documenting and maintaining exotic semantics nobody asked for. Backward-compatible to relax later.
- **`--component` on `closed`** — rejected. The archive is all-terminal; it holds no live components, so the filter would be always-empty or incoherent. This is the clean mirror of why `--closure` *does* span `closed` (corpus-wide, closed conducts) and `--component` does not.

## Consequences

- A new primitive lands in `query.clj` (e.g. `live-component`): it builds the undirected `:parent` ∪ `:deps` ∪ `:links` adjacency *restricted to live tickets* (closed non-conductive, broken refs dropped, visited-guarded — the same flood `connected-components` runs internally) and floods from the single seed, returning the component's member id-set **including a singleton `{seed}`**. This is the member-set `connected-components` computes but does not expose (its public map drops singletons). `closure-set` and `connected-components` are both untouched.
- `cli.clj` gains a `component-filter` beside `closure-filter`: when `opts` carries a resolved live `:component` seed, restrict `tickets` to the seed's live-component members (computed over the full corpus), a no-op otherwise. The closed-seed liveness check produces the fail-fast error where the corpus and `terminal-statuses` are in scope.
- `main.clj` gains a `resolve-component-filter!` mirroring `resolve-closure-filter!` (single-id partial resolution), wires it into the `list`/`ready`/`blocked` handlers, and rejects `--component` together with `--closure` at parse with a fail-fast guard. `closed` is untouched.
- `--component` membership is computed over the full live corpus and intersected before `filter-tickets`; `--limit` applies last. The `CC` column, JSON `cc` field, and all other output shapes are byte-unchanged — `--component` is a filter, not a visualization.
- `CONTEXT.md` gains a **Component filter** note under "Ticket relationships," cross-referencing the **Connected components** and **Closure** entries, with `_Avoid_` steering away from conflating it with `--closure`.
- Per the AGENTS.md skill-sync rule, `.claude/skills/knot/SKILL.md` documents `--component` on `list`/`ready`/`blocked` in the same commit that ships it.
- A future multi-seed union (`--component a,b`) or an emacs sticky-scope binding remains additive on this flag without revisiting this ADR.
