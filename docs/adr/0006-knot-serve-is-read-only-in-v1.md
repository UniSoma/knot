# knot serve is read-only in v1

`knot serve` ships as a viewer: `/api/ready`, `/api/in-progress`, `/api/blocked`, `/api/show/:id` shell out to `knot ... --json` and forward the envelope. There are no `POST`/`PATCH`/`DELETE` routes. A user who wants to mutate a ticket drops to the CLI or `knot.el`. A future reader looking at the namespace — given that both the CLI and `knot.el` are write-capable — would reasonably expect the browser surface to grow workflow verbs (`start`, `add-note`, `close`) at minimum. It doesn't, deliberately.

Two reasons gate writes out of v1. The first is **concurrency**: the prototype architecture is "shell out to `knot ... --json`," which is safe for idempotent reads but races against concurrent CLI and `knot.el` mutations on writes. Optimistic concurrency control on `:updated` is an explicit open follow-up (kno-01kqgqaxzx98); shipping write endpoints before that lands means shipping the race. The second is **CSRF surface**: a read-only loopback server with an origin allowlist is genuinely safe (any cross-origin `fetch` from a malicious tab is opaque and the data is local-only anyway). Accepting state-changing requests changes the threat model — same-machine pages, DNS rebinding, and any future shared-context bug become exploitable in ways "leak the existence of a ticket ID" is not. A per-launch token solves it, but that's a UX cost (printed URLs become long, copy/paste'd URLs break across launches) we don't want to pay before writes earn their keep.

The cost is that clicking around the panel eventually feels passive — you'll want to act on what you see. That's the right pressure to feel before designing the write surface, not a reason to design it now. Adding writes later is additive (new routes + middleware + token), not a breaking change, so this decision is genuinely reversible: a v2 ADR supersedes this one, the optimistic-concurrency ticket lands, and the write surface arrives with a real threat model attached.
